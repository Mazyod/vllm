## Why it hurts us (impact)

Structured / guided **JSON** output, generated by an agent that also uses
**reasoning (thinking)** and **MTP speculative decoding**, intermittently emits a
doubled opening brace — `{{` or `{"{`. The malformed wrapper breaks JSON parsing,
which surfaces downstream as (for example) `pydantic-ai "Exceeded maximum output
retries."`

The corruption is **wrapper-only**: the semantic content is intact, only the
leading structure is doubled. It affects native and tool output modes; prompted
mode (no grammar) is immune.

## Root cause (why the fix works)

A **three-way interaction**: MTP spec-decode, the reasoning parser, and
guided/structured decoding. Drop any one of the three and the corruption
disappears.

When the end-of-think marker and the first `{` of the answer land in the **same
speculative window**, the `grammar_bitmask` "reasoning ended mid-window" branch
sets `advance_grammar = False`. The `{` is committed to the output stream, but the
grammar FSM is **not advanced past it**. On the next step the FSM is still at
state 0 and emits another opener, producing `{{`. A single-request (concurrency 1)
trace corrupts identically, so concurrency only changes how often the window
straddles the boundary, not whether the bug exists. Fingerprint: a corrupt request
shows **no** grammar rollback at the commit step, while a clean request shows
`rollback(2)`.

There are **two coupled bugs**, and both must be fixed together (the grammar
change alone is inert):

1. **Detection timing** — the scheduler must pass the freshly sampled
   `new_token_ids` into `should_advance`, so it can see the tokens that crossed the
   reasoning boundary.
2. **Advance logic** — `should_advance` must return `True` and record the
   reasoning-end boundary for **all** constraint types, not only `STRUCTURAL_TAG`.

**The fix.** `scheduler.py` passes the tokens through:

```python
if new_token_ids and self.structured_output_manager.should_advance(
    request, new_token_ids=new_token_ids
):
```

`structured_output/__init__.py` extends the signature and advances the grammar
across the reasoning boundary for all constraint types:

```python
def should_advance(self, request, new_token_ids: list[int] | None = None) -> bool:
    ...
```

## Reproduce (cheap — Gemma-4-E2B rig)

`google/gemma-4-E2B-it` with its `-assistant` MTP draft reproduces the same bug
for a fraction of the cost of the 31B, because the mechanism lives in vLLM's
grammar bitmask and is **architecture independent**:

```bash
VLLM_USE_V2_MODEL_RUNNER=0 vllm serve google/gemma-4-E2B-it \
  --reasoning-parser gemma4 \
  --speculative-config '{"method":"mtp","model":<gemma-4-E2B-it-assistant>,"num_speculative_tokens":2}'
```

Drive it with structured (native) JSON requests that also enable thinking, at
concurrency around 16.

- **Stock `v0.25.1`:** roughly 75–100% of responses corrupt with `{{`.
- **With patch 0002:** 0% corrupt.

**Relevance check:** if stock (unpatched) shows 0% corruption here, the upstream
fix has landed — drop the patch.

## Validation (point-in-time)

On Gemma-4-E2B, applying both coupled changes moved corruption from
**75 / 100 / 96 / 91%** (at concurrency 8 / 16 / 24 / 32) to **0 / 0 / 0 / 0%**,
with no errors. Because the fault is in the shared grammar-bitmask path and not in
any model-specific code, the fix carries to the 31B production model.

## Ruled out (do not re-explore)

- **Parallel-vs-serial bitmask path** — the parallel threshold is 128 and the bug
  reproduces at concurrency 4, so that path is never taken.
- **Bitmask row-layout misalignment** — producer and consumer walk the same stride
  in the same order.

Both were ruled out during the investigation.
